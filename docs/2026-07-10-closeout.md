## Context & Motivation

The user dictates with **Wispr Flow** on a local Mac and works inside a **Parsec** remote-desktop session. Flow delivers text by simulating a paste (Cmd+V) into the locally-focused field. Inside Parsec this fails: Parsec is a video stream of the host, and the synthetic Cmd+V has its **Cmd modifier dropped** in transit, so only a lone `v` lands in the remote field.

Original request: *"How do I pass input through to Parsec while using Wispr Flow?"* — the observed symptom was *"I just get a 'v' that shows up in the text field after completing dictation."*

The request was refined several times:
1. First toward making Flow's paste work through Parsec.
2. Then: *"I want Wispr to push to clipboard, I can carry the final paste myself."*
3. Then a request for automation: *"is there something you can set up here to handle all that for me?"*
4. Then narrowed decisively: *"I can easily copy paste from the Mac to Parsec, so if you route the Wispr Flow completed dictations into clipboard I can just copy paste from there."*
5. Final shape, the user's own idea: *"find where wispr flow is storing last transcribed items, set up a script to watch when that changes and is stable and then push those inputs into the clipboard as they're complete."*

That last framing is the design that was built.

## Decisions Made

**Decision: Watch Flow's local database instead of injecting keystrokes.**
- Alternatives considered: (a) a Hammerspoon hotkey that types the clipboard into Parsec character-by-character (bypasses the modifier drop); (b) a fully-automatic clipboard watcher that types anything copied into a focused Parsec window; (c) relying on Flow's native "Copy last transcript" shortcut + a clipboard manager.
- Rationale: the user only needs the transcript *on the clipboard* (they paste themselves). Watching Flow's own transcript store is the cleanest, most reliable source and needs no Accessibility permission or keystroke simulation.
- Trade-offs: depends on Flow's internal, undocumented SQLite schema, which could change across Flow versions.

**Decision: Overwrite the system clipboard on every finalized dictation.**
- Rationale: exactly what the user asked for ("route completed dictations into clipboard").
- Mitigation: Maccy clipboard manager was installed so nothing is ever lost and history is scrollable. The automatic clipboard-watcher variant (option b above) was rejected as unsafe — it would type anything copied (e.g. a password) into the host.

**Decision: Publish as a standalone public repo with an installer.**
- Rationale: user asked to push it to its own public repo and document setup. Made the script env-var-configurable and added `install.sh`/`uninstall.sh` so it's reproducible on any Mac.

## What Was Built / Changed

Chronological narrative:

1. **Diagnosis.** Identified the modifier-drop as the cause of the stray `v`. Confirmed via Wispr Flow docs that Flow is officially unsupported on remote desktops and that its copy behaviors (Cmd+Ctrl+C "Copy last transcript", or dictating with no field focused) leave text on the clipboard, whereas auto-paste saves+restores the prior clipboard.
2. **Installed Maccy** (`brew install --cask maccy`) as the clipboard-history safety net.
3. **Located Flow's storage.** Flow is an Electron app (`com.electron.wispr-flow`). Data lives at `~/Library/Application Support/Wispr Flow/flow.sqlite`. The `History` table holds every dictation; relevant columns: `transcriptEntityId` (PK, updated in place), `asrText`, `formattedText` (final text), `status` (`raw_transcript` → `formatted` terminal, or `empty`), `numWords`, `timestamp`.
4. **Built the watcher** (`wispr-clip-watch.sh`): polls for the newest `status='formatted'` row and copies `formattedText` via `pbcopy`. Seeds "last seen" at startup so it doesn't clobber the clipboard with an old transcript.
5. **Packaged as a LaunchAgent** (`RunAtLoad` + `KeepAlive`), loaded via `launchctl bootstrap gui/$(id -u)`.
6. **First live test failed partially:** *"It caught the first one but not the next ones."* Debugging showed the process was alive and sitting in `sleep`, and fresh manual reads returned the newest row — but the running watcher's clipboard was stuck on the first dictation.
7. **Root-caused two live-environment bugs** (see Architecture) and shipped v3.
8. **Second live test passed:** all 5 consecutive dictations were captured in order, confirmed in the log.
9. **Published** to https://github.com/npezarro/wispr-flow-clipboard (initial commit `27b7ff8`). The push was first rejected by GitHub email-privacy protection; the commit was re-authored to the GitHub noreply address and pushed. SSH push also failed (no key on this host) → switched origin to HTTPS via `gh auth setup-git`.

## Architecture & Design

```
Wispr Flow (Electron)
   │ writes each dictation
   ▼
flow.sqlite  ──[History table, WAL mode]──┐
                                          │ polled every 0.5s (query-only)
                                          ▼
                              wispr-clip-watch.sh (LaunchAgent)
                                          │ pbcopy + ~1.2s re-assert
                                          ▼
                              macOS system clipboard  ──▶  user pastes into Parsec
                                          │
                                          ▼
                              Maccy (clipboard history / fallback)
```

**Two hazards specific to reading a live database the app is actively writing:**

1. **Stale WAL reads.** A `mode=ro` (`SQLITE_OPEN_READONLY`) connection can read a stale snapshot while Flow commits to the WAL. This is why the watcher "caught the first, not the rest" — it froze on an early snapshot. Fix: use a **normal, query-only** connection (`PRAGMA query_only=ON`), which participates in the WAL protocol correctly and reliably sees committed rows, while still never modifying Flow's data. (Reusable learning — see below.)

2. **Flow's clipboard restore race.** When you dictate into a focused field, Flow does save-clipboard → paste → **restore-clipboard**, overwriting whatever the watcher just set. Fix: after `pbcopy`, `put_clipboard()` re-checks `pbpaste` six times over ~1.2s and re-copies if something clobbered it, so the transcript wins.

Rows update **in place** (same `transcriptEntityId` across `raw_transcript` → `formatted`), so the watcher keys pushes on `status='formatted'` (terminal). Raw-only dictations are pushed after being stable for one poll cycle, to avoid grabbing text mid-formatting.
